📋 Prerequisites

Have these in place before clicking Onboard. Tap a tab below — most users only need to check tabs 1, 2, and the one matching their AI provider.

Microsoft 365 / Purview licenses on the tenant

The onboarder writes to Microsoft Purview and (optionally) Agent 365. Your tenant needs at least one of the following. Microsoft Learn: DSPM for AI ↗

Microsoft 365 E5 RECOMMENDED

Includes the Microsoft Purview Suite at no extra cost — covers Purview DLP, DSPM, and DSPM for AI out of the box. Nothing else to add.

Microsoft Purview Suite (formerly E5 Compliance) ALTERNATIVE

Add-on to M365 E3, Office 365 E3 + EMS E3, or M365 E5. The E5 Compliance SKU was renamed to Microsoft Purview Suite in 2025; existing E5 Compliance customers retain access. New standalone E5 Compliance purchases stopped after Sept 2025 — buy Purview Suite instead.

Microsoft Purview DSPM for AI (standalone add-on) ALTERNATIVE

If your tenant only needs the AI-governance scenarios, the DSPM-for-AI add-on alone is sufficient for what this onboarder creates. Devices monitoring 3rd-party AI sites also need the Microsoft Purview browser extension deployed.

Microsoft Agent 365 enrollment OPTIONAL

Only required if you tick Agent 365 blueprint or Agent 365 observability on the form. Tenant must be enrolled in the Agent 365 program. Microsoft Learn ↗

How to verify

Microsoft 365 admin center → Billing → Your products. Or for the signed-in user: Graph Explorer → GET /me/licenseDetails.

Sources: DSPM for AI overview · DSPM-for-AI considerations

Permissions for the signed-in user

The identity that runs az login is the one the onboarder acts as. They need:

Microsoft Entra ID
  • Application Developer — to create the app registration
  • Privileged Role Administrator or Global Administrator — to admin-consent the Graph permissions below
Microsoft Purview (DSPM for AI)
  • Compliance Administrator or Microsoft Purview Compliance Administrator — full management
  • View-only: Purview Security Reader, Purview Data Security AI Viewer, AI Administrator, or Purview Data Security AI Content Viewer
  • Access at purview.microsoft.com

Permissions granted to the new app (not held by you)

The onboarder requests these Application permissions on Microsoft Graph and admin-consents them as you:

PermissionPurpose
Content.Process.All processContent + contentActivities — required for the wrapper to run governed content through Purview.
ProtectionScopes.Compute.All protectionScopes/compute — required for the wrapper to evaluate which protection labels/policies apply.

If you lack the consent role the app + secret are still created; consent can be granted later from Entra → App registrations → <your app> → API permissions → Grant admin consent.

Sources: Purview DSPM-for-AI roles · Graph permissions reference

Provider credentials — what to set up before clicking Onboard

Only the card matching your chosen provider applies. Each card mirrors the form fields you'll fill in. Use Run diagnostics on the form to verify before submitting.

Google Vertex AI (Agent Engine / Reasoning Engine) END-TO-END TESTED

  1. In Google Cloud Console, create or pick a project and enable billing.
  2. Enable the Vertex AI API: gcloud services enable aiplatform.googleapis.com
  3. Grant your user (or service account) roles/aiplatform.user on the project. For deployments using a service account also grant roles/iam.serviceAccountUser.
  4. Build and deploy your ADK agent to Vertex Agent Engine. The deploy returns a resource name like projects/<num>/locations/us-central1/reasoningEngines/<id>.
  5. On your machine: gcloud auth application-default login.

Form fields: GCP project ID · region (e.g. us-central1) · Reasoning Engine resource name (paste the full projects/.../reasoningEngines/... string)

AWS Bedrock — Path A: Foundation model (Converse API) CODE-COMPLETE, NOT YET E2E-TESTED

  1. In the Bedrock console, switch to the region you want (e.g. us-east-1), open Model access, and explicitly request access to the model family you want (Anthropic Claude, Meta Llama, Amazon Titan, etc.). Some models require an org / use-case form.
  2. Grant the IAM identity (user / role) that will run the wrapper the action bedrock:InvokeModel on the model (or inference-profile) ARN. Add bedrock:InvokeModelWithResponseStream only if you'll use streaming (this onboarder does not).
  3. Configure credentials locally: aws configure (writes ~/.aws/credentials), OR set AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY / AWS_SESSION_TOKEN env vars, OR use an attached IAM role.
  4. Confirm the model ID: e.g. us.anthropic.claude-3-5-sonnet-20241022-v2:0. Many on-demand Anthropic models require the us. (or eu.) cross-region inference profile prefix — without it you'll get ValidationException: Invocation of model ID … with on-demand throughput isn't supported.

Form fields (Bedrock target = Foundation model): AWS region · Bedrock model ID · access key ID + secret access key (optional — leave blank to use the default credential chain) · session token (only for temporary STS creds)

AWS Bedrock — Path B: Deployed Bedrock Agent (InvokeAgent API) CODE-COMPLETE, NOT YET E2E-TESTED

  1. In the Bedrock console → Agents → Create agent. Pick a foundation model the agent should use, attach (optionally) knowledge bases + action groups, then click Prepare.
  2. After preparation, create an Alias for the agent (e.g. PROD pointing to a published version). Aliases let you promote new versions without changing this wrapper's config.
  3. Note the Agent ID (e.g. AGENT1234ABCD) and the Agent alias ID (e.g. TSTALIASID for the auto-created test alias, or your custom alias ID).
  4. Grant the caller / wrapper IAM principal the action bedrock:InvokeAgent on the agent alias ARN (NOT InvokeModel — that's a different permission for direct model calls). Example resource: arn:aws:bedrock:us-east-1:<account>:agent-alias/AGENT1234ABCD/TSTALIASID.
  5. Separately, the agent's own execution / service role (configured inside Bedrock when you created the agent) must already grant the agent permission to invoke its underlying foundation model and read any attached knowledge bases / action groups. This is set in the Bedrock console, not by the wrapper.
  6. Configure AWS credentials for the wrapper the same way as Path A (aws configure, env vars, or IAM role).

Form fields (Bedrock target = Deployed Bedrock Agent): AWS region · Bedrock Agent ID · Agent alias ID · AWS credentials as above

Session memory: the wrapper passes each chat's conversation_id as the Bedrock sessionId, so multi-turn conversations preserve server-side context per user. Set BEDROCK_SESSION_ID in the generated .env only if you want every chat to share one session (debug / single-user only).

Limitation: if your agent has an action group configured to return control to the caller, this wrapper does not implement action-group fulfillment and will raise a clear error. Either remove the action group from the agent or extend adapter.py to handle returnControl events.

Anthropic Claude (direct API) CODE-COMPLETE, NOT YET E2E-TESTED

  1. Sign in at console.anthropic.com and create (or join) a workspace.
  2. Under Plans & Billing, add a payment method (free trial credits are limited; sustained API use needs billing).
  3. Open API Keys → Create Key. Copy the sk-ant-… value immediately — it is shown once.
  4. Pick a current model id (e.g. claude-3-5-sonnet-20241022, claude-3-5-haiku-20241022) from the model list.

Form fields: Model · API key · Base URL (leave blank for https://api.anthropic.com; only set this when routing via a proxy like Helicone / Portkey / LiteLLM)

Azure OpenAI CODE-COMPLETE, NOT YET E2E-TESTED

  1. In the Azure portal, create an Azure OpenAI resource (region with capacity).
  2. Open Azure AI Foundry / Studio → Deployments and deploy a model (e.g. gpt-4o). Note the deployment name you choose.
  3. Grab the endpoint (https://<resource>.openai.azure.com/) and one of the keys from Keys & Endpoint.
  4. Pick an API version string (e.g. 2024-10-21 or latest from the API reference).

Form fields: Endpoint · Deployment name · API key · API version

OpenAI (platform.openai.com) CODE-COMPLETE, NOT YET E2E-TESTED

  1. Sign in at platform.openai.com, add a payment method under Billing.
  2. API keys → Create new secret key; copy the sk-… value once.
  3. Pick a model id (e.g. gpt-4o-mini, gpt-4.1) from the model list.

Form fields: API key · Model

Custom HTTP endpoint

  1. Stand up your own POST endpoint that accepts JSON and returns JSON.
  2. Decide the request shape (which JSON field holds the user prompt) and the response shape (JSONPath to extract the assistant reply).
  3. If the endpoint requires auth, prepare a bearer token or static header.

Form fields: Endpoint URL · Method · Prompt field name · Response JSONPath · optional Authorization header

Final pre-flight checklist

  • Tenant has Microsoft 365 E5 (or E5 Compliance / DSPM for AI) assigned
  • az login works and shows the right tenant + UPN
  • You have Application Developer in Entra (or someone on standby with consent rights)
  • You have Compliance Administrator in Purview
  • Python 3.10+ and the Azure CLI are on PATH
  • Provider credentials from Provider creds tab are ready
  • You know the local TCP port the wrapper will bind to (defaults to 8080)
  • For Vertex: gcloud auth application-default login already done
